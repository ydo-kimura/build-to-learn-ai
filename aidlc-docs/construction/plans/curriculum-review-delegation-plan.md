# Curriculum Review Remediation Delegation Plan

## Objective

`review.md` の指摘を再検証し、妥当なものだけを修正する。初学者向け教材としての正確性、説明とコードの整合性、演習の妥当性、画像・図・日英アセットの整合性を改善する。

今回は計画のみ作成する。修正作業はこの計画を基準に、担当ごとの分離された作業領域で実施する。

## 前提と保護方針

- 現在のワークツリーには、過去の画像・`public/`・英語版同期作業など未コミット変更がある。リセット、クリーン、広範囲な restore は行わない。
- 実作業の開始時に、現在の差分をスナップショット化して基準点を固定する。
- 各担当は担当範囲の本文・コード例・画像参照だけを編集し、他担当のファイルを変更しない。
- `aidlc-docs/audit.md`、`aidlc-state.md`、本計画書の進捗更新はコーディネーターが一元管理する。
- 日本語版を正とし、修正後に対応する英語版へ反映する。ただし英語版に固有の問題がある場合は別途記録する。

## 指摘の採用ルール

1. 現在のファイルを確認して、実在する不整合・誤説明・実行不能な例だけを必須修正とする。
2. 重大度は `review.md` のラベルをそのまま採用せず、学習への影響で再評価する。
3. 既に解消済み、事実誤認、または単なる好みの問題は「修正しない理由」を記録する。
4. 教材の改善提案（図の見た目、説明の追加、表現の好み）は、技術的な修正と分けて優先順位を下げる。

## 分担ワークストリーム

担当は同時進行可能だが、同じファイルを複数担当が編集しない。各担当は変更理由、対象指摘、検証結果を作業報告に残す。

### A. 基礎 ML: Unit 1〜9

対象: CH1-01〜CH1-10、CH1-12 のうち、本文・例示・演習に関するもの。

主な作業:

- One-Hot、評価指標、データ分割、前処理、モデル選択の説明と実装の整合性を確認。
- 初学者が実行できる最小例になっているか確認。
- Unit 5/9 の予告と実際の学習内容を整合させる。

編集範囲: `curriculum/unit01_*`〜`curriculum/unit09_*` と対応する日本語アセット。英語版は統合段階で反映。

### B. DL 基礎: Unit 10〜16

対象: CH2-01〜CH2-17。

主な作業:

- ニューラルネットワーク、PyTorch、損失関数、過学習、CNN、転移学習、Capstone の説明を確認。
- コードブロックの変数名、テンソル形状、実行順序を検証。
- 初学者に不要な高度化は追加せず、必要な前提説明を補う。

編集範囲: `curriculum/unit10_*`〜`curriculum/unit16_*` と対応する図・画像。

### C. NLP / Transformer: Unit 17〜21

対象: CH3-01〜CH3-18。

主な作業:

- TF-IDF、埋め込み、Self-Attention、位置エンコーディング、Transformer の説明とコードを照合。
- `sqrt(d_k)`、Q/K/V の役割、位置情報の加算位置など、学習上重要な数式・実装を優先修正。
- 図が本文・コードの概念と一致するか確認。

編集範囲: `curriculum/unit17_*`〜`curriculum/unit21_*` と対応する図・画像。

### D. LLM / Agent: Unit 22〜36

対象: CH4-02、CH4-03、CH4-05、CH4-07〜CH4-19。除外候補は上記ルールに従う。

主な作業:

- Tokenizer、Streaming、構造化出力、RAG、Tool Calling、Agent、評価、安全性の説明と実装を照合。
- API キーや外部サービスの前提を、実行可能なシミュレーション例と明確に区別。
- Unit 35/36 の基礎内容と Unit 22/23 の導入内容が重複しすぎないよう整理。
- LangChain などバージョン依存箇所は `package`/`requirements` と公式ドキュメントを確認して判断。

編集範囲: `curriculum/unit22_*`〜`curriculum/unit36_*` と対応する図・画像。

### E. Capstone / 応用: Unit 37〜42

対象: CH5-01〜CH5-16。

主な作業:

- 評価、セキュリティ、構造化抽出、マルチエージェントの説明と実装の整合性を確認。
- Unit 37 のタイトルと実際の評価内容を整合。
- Unit 39 の JSON フェンス処理を実際の入力形式に合わせる。
- Unit 42 のエージェント数、役割、図、実装、解答を一つの構成に統一。

編集範囲: `curriculum/unit37_*`〜`curriculum/unit42_*` と対応する図・画像。

### F. 横断品質・アセット

担当: コーディネーター。

主な作業:

- 以前の方針どおり、ページ本文と重複する「タイトルだけ」の日本語ヒーロー画像変更を整理する。
- `assets/`、`public/assets/`、英語版アセットの参照関係を確認し、言語ごとのデプロイ元を一貫させる。
- 画像内の説明文・図については、単なるタイトル変更と学習内容の翻訳を分けて判断する。
- 全担当の修正を統合し、日英の構造・参照・画像パスを確認する。

編集範囲: `assets/`、`public/assets/`、`en/assets/`、`public/en/assets/`、各言語の画像参照。本文担当の Markdown は直接編集しない。

## 実施順序

1. **基準化**: 現在の差分一覧、ブランチ、既存の画像・英語版変更を記録する。
2. **並行修正**: A〜E をファイル範囲ごとに分離して実施する。
3. **横断整理**: F がアセット参照、タイトル画像、日英対応を整理する。
4. **統合レビュー**: 指摘ごとに「修正済み / 保留 / 却下」を再確認する。
5. **自動検証**: ビルド、図、Python ブロック、Markdown リンク、画像パス、差分 whitespace を検証する。
6. **教材レビュー**: 初学者の学習順序、説明・例示・演習・図の一貫性を章単位で確認する。
7. **記録**: `audit.md`、`aidlc-state.md`、本計画のチェックボックスを更新する。
8. **コミット準備**: 変更内容を整理し、ユーザー確認後にコミット・プッシュ・PR作成へ進む。

## 検証項目と完了条件

- `npm run docs:build` が成功する。
- `python3 scripts/verify_curriculum_diagrams.py` が成功する。
- Python fenced block の構文検証が成功する。
- Markdown のローカルリンクと画像参照が解決する。
- 日英両方で Unit 0〜42 の構成、ナビゲーション、画像パスが整合する。
- Unit 20 の attention 数式・コード、Unit 29 の関数名・参照、Unit 39 の JSON 処理、Unit 42 のエージェント構成を個別に再確認する。
- `git diff --check` が成功する。
- 各採用指摘に、修正箇所と検証結果が対応付く。
- 保留・却下した指摘には、現行コードに基づく理由が残る。

## 分担上の制約

- エージェントは別 worktree または別ブランチで実行し、担当範囲外の差分を取り込まない。
- 共有ファイルの競合を避けるため、ナビゲーション、監査記録、状態ファイル、最終アセット同期は統合担当が行う。
- 実装前に計画を変更する必要が出た場合は、対象指摘と編集範囲を本計画に追記してから着手する。
